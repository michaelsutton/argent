

drafting a route tree algo. 

we have a a directed graph G = (V, E)

V is the set of actors. E has a directed edge (u, v) if u has a possible emit transition to v. 
as std n = |V|, m = |E|

the result of the algo is: 
1. a tree of hashes "the commit tree" (actually a forest, bcs we can have multiple roots. eg if we keep all templates open at each actor, we have n trees) 
   where each actor has a leaf 
2. a per actor partial-cut of the tree (cut can be partial if that actor cannot reach some leafs; the main point of cut here: no tree nodes that are ancestors or descendants one of the other)

the challenge is that the construction of 1 is highly influenced by 2, so we basically need to produce both as a single logical unit. 

informal cost function / optimization target:
- when an actor adds a field (eg for holding a route commit), it directly influences its redeem script size (and hence increasing the sigscript size and the p2sh hashing)
- that implies we should aim to compress such fields (go to higher levels in the commit tree/forest)
- however that is valid only if there are no actor-local freq usages of that field
- if the field needs to be opened at each entry call, then the size and hashing costs are merely shifted to another nested preimage witness provided by an additional sig arg. 
- hence, compression only makes sense if openings (maybe expansions are a better name) are rare and are the exception. 

relaxation/heuristic:
- assume that actor graphs usually have a locality pattern and add the constraint that each route transition only expands/packs a single level (it can expand/pack multiple tree nodes, but it supposedly never expands/packs more than 1 tree level in a single transition)
- the above might not be a trivial property for general graphs (where there might be 2 paths between (u,w) one very long and one very short), so i think we should first focus on an easy and intuitive algo, and then reason about its properties 

sketch idea:
- identify strong components in the graph that freq transition to one another and should remain identical in state and tree cut. 
- one such obvious component is a route family (defined by the user as well; not sure how strongly current code relates the two)
- iirc the family is identified as a group of actors sharing state layout (user state) and which for each actor there exists another actor in the group which has a direct edge to/from it
- we also defined the entries to the family. to be recalled. 

basically, this family only means that all its members hold a full table of each other and each share the same tree cut, but otherwise it does not tell us how to process the rest of the nodes. what should be the basic guiding principal other than families.
